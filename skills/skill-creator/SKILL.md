# Skill Creator: 나만의 스킬 만들기

> **핵심**: 대화만으로 나만의 스킬을 만든다.
>
> 어떤 업무를 스킬로 만들고 싶은지 파악하고 →
> 워크플로우를 같이 설계하고 →
> 스킬 파일을 생성해서 → 바로 테스트한다.

## 실행 흐름 요약

```
Phase 1 → Phase 2 → Phase 3 → Phase 4
의도 파악   인터뷰+설계   스킬 생성   테스트+반복
```

---

## Phase 1: Intent Capture — 의도 파악

사용자 요청의 구체성에 따라 분기한다.

### 구체적 요청 ("주간보고 스킬 만들어줘")

추가 질문 없이 빠른 확인만 진행한다:
```
"주간보고" 스킬을 만들어드릴게요!
몇 가지만 확인할게요.
```
→ Phase 2로 이동

### 막연한 요청 ("스킬 만들어줘", "자동화", "뭘 할 수 있어")

ToolSearch로 사용 가능한 도구를 파악한 뒤, 예시를 제시한다:
```
어떤 업무를 스킬로 만들고 싶으세요?

예를 들면:
• {사용 가능한 도구 기반 예시 2~3개}
• 또는 떠오르는 아이디어가 있으면 말씀해주세요!
```

> **예시 선택 기준** (사용 가능한 도구에 따라):
> - Jira+Confluence: "Jira 티켓을 정리해서 주간보고 위키에 작성", "스프린트 리뷰 자료 생성"
> - Slack+Confluence: "Slack 대화를 위키 회의록으로 정리", "채널 논의 요약"
> - Jira+Slack: "Jira 현황을 Slack에 공유", "Slack 논의를 Jira 티켓으로 변환"
> - 도구 없음: "매주 보내는 이메일 본문 작성", "회의 안건 정리", "일일 업무 체크리스트"

AskUserQuestion으로 답변을 받는다.

### 수정 요청 ("스킬 수정해줘")

기존 스킬을 검색하여 수정 흐름으로 진행한다:
1. `skills/` 디렉토리에서 해당 스킬을 찾는다
2. 현재 SKILL.md 내용을 읽는다
3. 수정 사항을 확인한다
4. SKILL.md를 업데이트한다 (Edit 도구로 수정)

---

## Phase 2: Interview + Design — 인터뷰 + 설계

**목적**: 스킬의 목적, 동작, 입출력을 파악하고 워크플로우를 설계한다.

### Step 1: 구체화 질문

한 번에 1~2개씩만 질문한다 (폭격 금지).

확인 항목:
- **스킬 이름**: 무엇이라 부를지
- **하는 일**: 한 줄로 설명
- **입력**: 사용자가 제공하는 것 또는 "없음 (자동)"
- **출력**: 결과물 형태 (위키, PPT, Excel, 화면 등)
- **사용 도구**: 필요한 연결 도구

### Step 2: 요약 확인

인터뷰 결과를 정리하여 확인한다:
```
정리해볼게요.

만들 스킬:
  이름: {스킬명}
  하는 일: {한 줄 설명}
  입력: {사용자가 제공하는 것 또는 "없음 (자동)"}
  출력: {결과물 형태}
  사용 도구: {필요한 도구}

이렇게 맞나요?
```

AskUserQuestion: "맞아요" / "수정할래요"
- 맞음 → Step 3로
- 수정 → 수정 후 다시 확인

### Step 3: 워크플로우 초안

인터뷰 결과를 바탕으로 워크플로우를 자연어로 설계한다:
```
이런 순서로 동작하게 만들어볼게요:

"{스킬명}" 스킬이 실행되면:
1. {Step 1 — 자연어로 설명}
2. {Step 2}
3. {Step 3}
...

이 순서로 괜찮을까요? 빠진 단계나 추가할 게 있으면 말씀해주세요.
```

AskUserQuestion: "좋아요" / "수정할래요"
- 좋아요 → Phase 3으로
- 수정 → 수정 후 다시 확인

---

## Phase 3: Write SKILL.md — 스킬 생성

**목적**: 설계된 워크플로우를 스킬 파일로 생성한다.

### Step 1: 이름 충돌 확인

기존 `skills/` 디렉토리에 동일 이름 스킬이 있는지 확인한다.

**시스템 스킬 보호**: `connector`, `automation`, `onboarding`, `skill-creator` 이름은 시스템 스킬이 사용 중이므로 사용자 스킬로 생성/업데이트할 수 없다. 이 이름이 요청되면 "이 이름은 시스템에서 사용 중이에요. 다른 이름을 사용해주세요."라고 안내하고 다른 이름을 제안한다.

이름이 겹치는 경우 (시스템 스킬 제외):
```
"{스킬명}" 스킬이 이미 있어요!

① 기존 스킬을 업데이트하기 (새 버전으로 수정)
② 다른 이름으로 만들기
③ 기존 스킬을 먼저 확인하기

어떻게 할까요?
```

### Step 2: 스킬 파일 생성

`references/skill-guide.md`를 참조하여 스킬 파일을 생성한다.

생성 위치: `skills/{스킬명}/SKILL.md`

**작성 원칙:**
- frontmatter: name, description, triggers 포함
- triggers는 스킬명, 띄어쓰기 변형, "~해줘" 변형 등 최소 3개 자동 생성
- `user-invocable: true` 기본 설정
- description은 "무엇을 하는지 + 언제 사용하는지" 포함
- 본문은 imperative form (동사 시작 지시문)
- 자연어로 워크플로우 설명 (MCP 함수명 금지)

### Step 3: 생성 결과 안내

```
스킬이 만들어졌어요!

  이름: {스킬명}
  동작: {한 줄 요약}
  사용법: "{스킬명}"이라고 말하면 실행돼요
```

## Phase 4: Test + Iterate — 테스트 + 반복

**목적**: 생성된 스킬을 즉시 실행하여 동작을 확인한다.

### Step 1: 테스트 실행

```
방금 만든 "{스킬명}" 스킬을 테스트해볼게요!
```

생성된 SKILL.md 파일을 Read 도구로 읽고, 해당 지시에 따라 워크플로우를 실제로 실행한다.
- 각 단계마다 진행 상황을 표시한다
- 결과물이 있으면 미리보기를 보여준다

### Step 2: 피드백

테스트 완료 후:
```
테스트 완료!

결과가 괜찮으세요?

① 좋아요, 이대로 쓸게요
② 좀 수정하고 싶어요 → 어떤 부분을 수정할지 듣고 스킬 파일 수정 후 다시 테스트
```

AskUserQuestion으로 확인.
- ① → Step 3 (완료)
- ② → 수정 사항 확인 후 스킬 파일 수정 → 다시 테스트

> 수정이 2회 이상 반복되면: "테스트가 잘 안 되고 있네요. 일단 스킬을 저장하고, 나중에 수정할 수도 있어요." 안내 후 "일단 저장하기" / "계속 수정해보기" 중 선택.
> 도구 연결 문제로 실패하면 connector 안내를 추가로 제공한다.

### Step 3: 완료

```
"{스킬명}" 스킬이 완성되었어요!

  사용법: "{스킬명}"이라고 말하면 실행돼요
  수정: "{스킬명} 스킬 수정해줘"라고 말하면 돼요
```

추가 스킬 제안:
```
다른 스킬도 만들어볼까요?

① 하나 더 만들기 → Phase 1로 돌아감
② 오늘은 여기까지
```

AskUserQuestion으로 선택.
- ① → Phase 1로
- ② → 종료 안내:
  ```
  나중에 만들고 싶으면 "스킬 만들어줘"라고 말하면 돼요!
  ```

## 행동 규칙

### 대화 톤
- 비개발자/개발자 state를 확인하지 않는다 — 사용자 대화 톤에 자연스럽게 맞춘다
- 기술 용어는 상황에 따라 자연스럽게 조절한다

### 기술 용어 변환
| 기술 용어 | 사용자에게 보이는 표현 |
|-----------|---------------------|
| SKILL.md | (보여주지 않음) |
| skills/ 디렉토리 | (보여주지 않음) |
| YAML frontmatter | (보여주지 않음) |
| MCP 서버 | 연결된 도구 |
| mcp__* | (보여주지 않음) |

### 인터뷰 규칙 (Phase 2)
- **열린 질문으로 시작**, 사용자 답변에 따라 구체화
- 한 번에 질문 1~2개만 (폭격하지 않기)
- 사용자가 막히면 **사용 가능한 도구 기반 예시**로 전환
- 이미 구체적이면 빠른 확인만 하고 넘어가기

### 스킬 생성 규칙 (Phase 3)
- `references/skill-guide.md`의 가이드를 반드시 참조
- 기존 `skills/` 디렉토리에 동일 이름 스킬이 있는지 확인
- **MCP 함수명을 스킬 파일에 쓰지 않는다** (자연어만 사용)

### 에러 처리
- 기술적 에러 → 사용자 친화 메시지로 변환
- 연결 안 된 도구가 필요한 스킬 → "connector" 안내 + 대안 제안
