# Skill Creator: 나만의 스킬 만들기

> **핵심**: 대화만으로 나만의 스킬을 만든다.
>
> 어떤 업무를 자동화하고 싶은지 인터뷰하고 →
> 워크플로우를 같이 설계하고 →
> 스킬 파일을 생성해서 → 바로 테스트한다.
> 코드를 몰라도, 스킬 포맷을 몰라도 된다.

## automation 스킬과의 차이

- **automation**: 연결된 도구를 활용한 "자동화 체험"에 집중. 워크플로우를 실행해보고, 원하면 스킬로 저장.
- **skill-creator**: "스킬 파일 생성" 자체에 집중. 더 구조적인 인터뷰와 설계 확인을 거쳐, 품질 높은 스킬 파일을 만든다.

사용자가 "자동화 만들기"를 원하면 automation, "스킬 만들기"를 원하면 이 스킬을 사용한다.

## 실행 흐름 요약

```
Phase 0 → Phase 1 → Phase 2 → Phase 3 → Phase 4 → Phase 5
환경 확인   인터뷰    설계 확인   스킬 생성   테스트     완료
 (자동)   (대화)    (구조화)    (파일)    (실행)    (안내)
```

---

## Phase 0: 환경 확인

**목적**: 사용자 역할과 연결된 도구를 파악한다.

1. Read 도구로 `~/.claude/skills/smilegate-ai-tools/state.json`을 읽는다.
2. `onboarding.role` 값을 확인한다:
   - `"developer"` → 개발자 모드 (기술 디테일 포함 가능)
   - `"non-developer"` 또는 값 없음 → 비개발자 모드 (기술 용어 숨김)
3. `connector.connected_services` 배열을 확인한다:
   - 연결된 서비스 목록을 파악한다 (사용 가능한 도구)
   - 연결된 서비스가 **0개**이면:
     ```
     아직 연결된 업무 도구가 없어요.
     스킬을 만들려면 먼저 도구를 연결하는 게 좋아요!
     "커넥터" 또는 "/connector"라고 말하면 바로 연결할 수 있어요.

     도구 없이도 스킬을 만들 수 있긴 하지만,
     도구를 연결하면 훨씬 강력한 스킬을 만들 수 있어요.
     ```
     AskUserQuestion: "도구 연결 후에 만들래요" / "도구 없이 만들래요"
     - 도구 연결 → 종료하고 connector 안내
     - 도구 없이 → Phase 1로 진행

## Phase 1: 인터뷰 — "어떤 스킬을 만들고 싶으세요?"

**목적**: 사용자가 만들고 싶은 스킬의 목적과 동작을 파악한다.

### Step 1: 시작 질문

**사용자가 이미 구체적으로 요청한 경우** (예: "주간보고 스킬 만들어줘"):
- 추가 질문 없이 빠른 확인만 진행한다:
  ```
  "주간보고" 스킬을 만들어드릴게요!
  몇 가지만 확인할게요.
  ```
  → Step 2 구체화 질문으로 이동

**사용자가 막연하게 요청한 경우** (예: "스킬 만들어줘"):
- 열린 질문으로 시작한다:
  ```
  어떤 업무를 스킬로 만들고 싶으세요?

  예를 들면:
  • {연결된 도구 기반 예시 2~3개}
  • 또는 떠오르는 아이디어가 있으면 말씀해주세요!
  ```

> **예시 선택 기준** (연결된 도구에 따라):
> - Jira+Confluence: "Jira 티켓을 정리해서 주간보고 위키에 작성", "스프린트 리뷰 자료 생성"
> - Slack+Confluence: "Slack 대화를 위키 회의록으로 정리", "채널 논의 요약"
> - Jira+Slack: "Jira 현황을 Slack에 공유", "Slack 논의를 Jira 티켓으로 변환"
> - 도구 없음: "데이터 정리", "보고서 작성", "파일 변환"

AskUserQuestion으로 답변을 받는다.

### Step 2: 구체화 질문

사용자 답변의 구체성에 따라 분기한다.

**구체적 답변** → 바로 정리:
> "Jira에서 내 티켓 가져와서 주간보고 위키에 써주는 스킬"
→ 핵심 요소만 확인하고 Phase 2로

**막연한 답변** → 한 번에 1~2개만 질문 (폭격 금지):
> "보고서 관련?"
→ "어떤 보고서인가요? (예: 주간보고, 월간보고, 프로젝트 현황)"
→ "데이터는 어디서 가져올까요? (예: Jira, Slack, 직접 입력)"
→ "결과물은 어떤 형태가 좋으세요? (위키, PPT, Excel, 화면)"

### Step 3: 요약 확인

인터뷰 결과를 정리하여 확인한다:
```
정리해볼게요.

만들 스킬:
  이름: {스킬명}
  하는 일: {한 줄 설명}
  입력: {사용자가 제공하는 것 또는 "없음 (자동)"}
  출력: {결과물 형태}
  사용 도구: {연결된 도구 중 필요한 것}

이렇게 맞나요?
```

AskUserQuestion: "맞아요" / "수정할래요"
- 맞음 → Phase 2로
- 수정 → 수정 후 다시 확인

## Phase 2: 설계 확인 — "이런 순서로 동작할 거예요"

**목적**: 스킬의 워크플로우를 구체적으로 설계하고 사용자와 확인한다.

### Step 1: 워크플로우 초안

인터뷰 결과를 바탕으로 워크플로우를 설계한다:
```
이런 순서로 동작하게 만들어볼게요:

"{스킬명}" 스킬이 실행되면:
1. {Step 1 — 자연어로 설명}
2. {Step 2}
3. {Step 3}
...

이 순서로 괜찮을까요? 빠진 단계나 추가할 게 있으면 말씀해주세요.
```

**비개발자**: 각 단계를 자연어로 쉽게 설명 (기술 용어 없이)
**개발자**: 기술 디테일 포함 가능 (사용할 API, 데이터 형식 등)

AskUserQuestion: "좋아요" / "수정할래요"

---

## Phase 3: 스킬 생성 — "스킬을 만들게요"

**목적**: 설계된 워크플로우를 스킬 파일로 생성한다.

### Step 1: 스킬 파일 생성

`references/skill-guide.md`를 참조하여 스킬 파일을 생성한다.

생성 위치: `skills/{스킬명}/SKILL.md`

**작성 원칙:**
- frontmatter: name, description, triggers 포함
- description은 third-person 형식 ("This skill should be used when..." 또는 한국어로 설명)
- 본문은 imperative/infinitive form (동사 시작 지시문)
- 자연어로 워크플로우 설명 (MCP 함수명 금지)
- 스킬명은 한글 권장 (비개발자 친화)

### Step 2: 생성 결과 안내

**비개발자에게는 기술 디테일을 숨긴다:**
```
스킬이 만들어졌어요!

  이름: {스킬명}
  동작: {한 줄 요약}
  사용법: "{스킬명}"이라고 말하면 실행돼요
```

**개발자에게는 파일 위치도 안내한다:**
```
스킬이 만들어졌어요!

  이름: {스킬명}
  파일: skills/{스킬명}/SKILL.md
  동작: {한 줄 요약}
  사용법: "{스킬명}"이라고 말하면 실행돼요
```

---

## Phase 4: 테스트 — "바로 테스트해볼게요"

**목적**: 생성된 스킬을 즉시 실행하여 동작을 확인한다.

```
방금 만든 "{스킬명}" 스킬을 테스트해볼게요!
```

스킬의 워크플로우를 실제로 실행한다.
- 각 단계마다 진행 상황을 표시한다
- 결과물이 있으면 미리보기를 보여준다

테스트 완료 후:
```
테스트 완료!

결과가 괜찮으세요?

① 좋아요, 이대로 쓸게요
② 좀 수정하고 싶어요 → 어떤 부분을 수정할지 듣고 스킬 파일 수정 후 다시 테스트
```

AskUserQuestion으로 확인.
- ① → Phase 5
- ② → 수정 사항 확인 후 스킬 파일 수정 → 다시 테스트

---

## Phase 5: 완료 — "다음부터 이렇게 쓰세요"

**목적**: 사용법을 안내하고, 추가 스킬 생성을 제안한다.

### Step 1: 완료 리포트

```
"{스킬명}" 스킬이 완성되었어요!

  사용법: "{스킬명}"이라고 말하면 실행돼요
  수정: "{스킬명} 스킬 수정해줘"라고 말하면 돼요
```

### Step 2: 추가 스킬 제안

```
다른 스킬도 만들어볼까요?

① 하나 더 만들기 → Phase 1로 돌아감
② 오늘은 여기까지
```

AskUserQuestion으로 선택.
- ① → Phase 1로 (Phase 0 건너뜀, 환경 정보 재활용)
- ② → 종료 안내:
  ```
  나중에 만들고 싶으면 "스킬 만들어줘"라고 말하면 돼요!
  ```

## 행동 규칙

### 인터뷰 규칙 (Phase 1)
- **열린 질문으로 시작**, 사용자 답변에 따라 구체화
- 한 번에 질문 1~2개만 (폭격하지 않기)
- 사용자가 막히면 **연결된 도구 기반 예시**로 전환
- 이미 구체적이면 빠른 확인만 하고 넘어가기

### 설계 규칙 (Phase 2)
- 워크플로우는 **자연어로 설명** (비개발자에게는 기술 용어 금지)
- 입력/처리/출력을 한눈에 보이게

### 스킬 생성 규칙 (Phase 3)
- `references/skill-guide.md` 의 가이드를 반드시 참조
- 기존 `skills/` 디렉토리에 동일 이름 스킬이 있는지 확인
- 이름 충돌 시: 업데이트 / 다른 이름 / 기존 확인 중 선택
- **MCP 함수명을 스킬 파일에 쓰지 않는다** (자연어만 사용)
- 스킬 이름은 한글 권장: `주간보고`, `회의록`, `티켓생성`

### 비개발자 배려 규칙
- 기술 용어 변환 (automation 스킬과 동일):

| 기술 용어 | 사용자에게 보이는 표현 |
|-----------|---------------------|
| SKILL.md | (보여주지 않음) |
| skills/ 디렉토리 | (보여주지 않음) |
| YAML frontmatter | (보여주지 않음) |
| MCP 서버 | 연결된 도구 |
| mcp__* | (보여주지 않음) |

### 에러 처리
- 기술적 에러 → 사용자 친화 메시지로 변환
- 연결 안 된 도구가 필요한 스킬 → "connector" 안내 + 대안 제안
